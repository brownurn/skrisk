# SK Risk Kickoff Notes

## Request Summary

The project was started as `~/code/skrisk` to create a “Skills Risk” system analogous to `cwsrisk`, but focused on AI agent skills instead of Chrome extensions. The requested outcome was a collector and analysis platform that can scrape top skills, mirror their source repos, look for malware and exfiltration behavior, classify risk, retain historical versions, re-scan every 72 hours, and present the results in a public-facing dashboard.

## Repositories Reviewed

- `cwsrisk`: primary implementation template for discovery, historical storage, analysis, enrichment, and dashboard/API
- `mewhois`: shared WHOIS and domain-intel service to reuse later
- `meip`: shared IP/DNS/ASN service to reuse later
- `melurna_v2`: downstream analytics/dashboard host pattern for future integration
- nearby Melurna repos for general product context

## External Research Inputs

- `skills.sh` docs, sitemap, and audits pages
- Cisco AI Defense `skill-scanner`
- Snyk ToxicSkills reporting
- Mitiga research on silent codebase exfiltration via skills
- Academic work such as `Skills in the Wild`

## Decisions Reached

- Build SK Risk as a new standalone scanner/source system
- Sequence v1 as `collection-first`
- Scope v1 to `skills.sh` plus linked GitHub repos
- Re-scan each tracked repo every 72 hours
- Reuse the `cwsrisk` pattern of versioned snapshots, deep analysis, and public API/dashboard
- Keep partner verdicts visible, but maintain SK Risk’s own evidence-first risk model

## Review-Fix Cycle

An independent review after the first implementation identified four major operational issues that were then fixed in-tree:

- unchanged rescans were blocked by a uniqueness constraint on skill snapshots
- mirrored repos were not advancing to new commits on refresh
- repo snapshots were being created once per skill instead of once per repo
- one broken skill load could abort the entire registry sync

The current tests include explicit regressions for those cases.

## Current Limitations

- source repo URLs are still assumed when the registry does not expose them directly
- `mewhois`, `meip`, and third-party intel adapters are not wired yet
- the production Postgres/Timescale deployment path remains a next-phase task
